# How to make Himmy much better — strategic brief
_2026-06-23, from a 6-agent multi-lens analysis (product depth · agent intelligence · reliability/trust · distinctiveness), adversarially verified against the code._

## Headline
Himmy is a beautiful reader + chatbot, but its one un-cloneable asset — YOUR papers, highlights, notes and history — currently powers almost nothing. The single most important move is to close that loop: make everything you write and mark up feed back into what Himmy knows and recommends, and protect that growing personal corpus with a backup that actually saves it.

## Honest verdict
Himmy is genuinely strong where most apps are demos: the PDF Reader with real highlights, the streaming ⌘K assistant with durable human-in-the-loop approvals, semantic RAG over your library (real fastembed, not a hash fallback), auto-injected memory, and a working scheduler are all built and verified in code — not mockups. Where it falls short of "much better" is that these are disconnected silos that don't compound: your highlights and notes are invisible to the assistant's brain (library.py rag_records reads neither), recommendations key off a hand-typed keyword box instead of what you actually read (news.py), the RAG index re-embeds from scratch every launch, and most dangerously, the "Back up" feature silently saves only your papers while leaving every conversation, memory, task and routine behind — with a restore that deletes your live files before checking the backup is even valid.

## Recommended next
Build the whole-workspace backup and safe restore first (Big Move 1), with a round-trip restore test. It is the one move with no upside ceiling and a real downside today — every other improvement makes Himmy remember more about you, and all of that new value lands in stores the current 'backup' silently drops and the current 'restore' can destroy. Protect the corpus before you spend effort making it deeper, then immediately move to persisting and enriching the index (Big Move 2), which everything else builds on.

## Big moves (step-changes, ordered by leverage)

### 1. Whole-workspace backup + safe restore — protect the data before you grow it — effort M / impact high
**Why it matters:** The feature you sell as 'protect your data' is a data-loss trap: it saves only your PDF library and silently leaves out every conversation, everything Himmy has learned about you, your tasks and your routines. Worse, restoring deletes your live files BEFORE checking the backup file is good — so one corrupt backup wipes everything. As every other improvement below makes Himmy remember more about you, this unprotected, destructively-restored state becomes the very thing you can least afford to lose. This must be fixed first, because you should not grow data you cannot safely save.

**What it takes:** Rewrite library.backup() (src/himmy_app/library.py:558, today it only zips library.db + library_files/) to capture the whole .scholar-desk/ directory — conversations.db, memory.db, tasks.db, routines.db, inbox.db, approvals.db, storage.db, usage.json (all confirmed present on disk and excluded today) — using SQLite's safe-copy so you never zip a half-written database; skip only the regenerable papers_cache.db. Rewrite restore() (line 571) to validate the zip and snapshot the current state to a .pre-restore folder BEFORE touching live files (today line 584 unlinks first). Add a manifest with item counts so restore can sanity-check. Critically, add a round-trip restore test in tests/ — a backup rewrite shipped without one is itself a data-loss risk.

### 2. Feed your highlights and notes into Himmy's brain — on a persistent index — effort M / impact high
**Why it matters:** Himmy reads the raw text of your papers but is blind to YOUR thinking about them. The lines you highlighted and the notes you wrote in the margins — the highest-value, hand-curated signal in your whole library — never reach the assistant. So it can tell you what a paper says, but not what YOU found important, and it can't connect your annotations across papers. Pair this with persisting the index to disk and Himmy stops being a search box over PDFs and starts being an assistant that actually knows your research — and it's instantly smart after every restart instead of going blind for 2-3 minutes while it silently re-reads everything.

**What it takes:** Two wired-in-order changes. First, swap the in-memory index for disk: papers_rag.py:184 passes backend=None — point it at the framework's SqliteKnowledgeBackend (it exists at himmy/services/knowledge/sqlite_backend.py, already does content-hash dedup and incremental re-ingest, just unused), so embeddings survive restarts and only changed papers re-embed. Second, extend library.py:184 rag_records() to also SELECT the notes column and join the highlights table (both exist in the schema, lines 109/112), and in papers_rag _record_text append each paper's notes and highlight texts, tagged so the assistant can cite YOUR margin notes. Bust the text cache when annotations change. Verify the persist path is safe across the separate thread/event-loop that deep research spins up.

### 3. Make your corpus the recommendation brain — retire the keyword box — effort M / impact high
**Why it matters:** This is the moat move. Right now a brand-new install and a six-month-deep install behave almost identically, because For You news and arXiv recommendations are driven entirely by a keyword list you type by hand (news.py _for_you/_arxiv/recommendations) — your actual reading history and highlights drive zero personalization. The moment the feed is built from what you read and mark up, a competitor literally cannot reproduce it, and the product gets sharper the more you use it. That 'the longer I use it the more it becomes irreplaceably mine' feeling is the only durable advantage over Notion/Zotero/ChatGPT, and it's one wire away once the corpus is indexed on disk (move 2).

**What it takes:** Replace the hand-typed interests path in news.py (lines 309/358/383) with a corpus-derived signal: reuse the embedder and persisted vectors from move 2 to build a 'what this user reads' centroid (weighting recent highlights higher), generate arXiv/news candidate queries from your top library terms, and re-rank candidates by similarity to that centroid. Keep the keyword box as an optional override and dedup against papers already in your library. Mind the cost: candidate-embedding and ranking should lean on the local fastembed model, not paid API calls, since the assistant runs on a metered model.

### 4. A real Notes surface — the missing 'thinking space' — feeding the same index — effort L / impact high
**Why it matters:** Himmy has a great place to READ (Reader) and to CHAT (⌘K), but nowhere to WRITE. The only notes live trapped per-PDF inside the reader; you can't keep a standalone research note, a daily log, or a literature-review draft. For a tool meant to displace Obsidian/Notion this is the single biggest product hole — it's why Himmy is a polished app you visit, not the workspace you live in. Done right, your notes also flow into the same brain from move 2, so the place you think and the assistant that knows your thinking are the same system.

**What it takes:** Add a Notes nav tab and a backend notes store mirroring the existing tasks pattern (server.py task-store style: a himmy SQLite store + CRUD endpoints + an api.notes client), reusing the markdown rendering already in the app. Index notes into the persistent RAG backend from move 2 so 'ask my library' covers them, and add @-mention linking to a Paper or SavedArticle id. The editor surface is the bulk of the work; the backend closely mirrors the tasks and saved-news code that already exists. (This is also the prerequisite for a genuinely useful global ⌘F search across papers, highlights, notes, articles and tasks — a strong quick-follow once Notes exists.)

### 5. Make automations trustworthy and visible: fire when asleep, and show what Himmy did — effort L / impact med
**Why it matters:** Two trust gaps undercut the 'assistant' promise. First, the scheduler lives inside the app process — closing the window is fine on Mac (the app stays in the dock), but a full Quit or the Mac sleeping silently stops your 6:30 briefing, and when it catches up it fires at a random time, which feels broken. Second, there's no durable record of what the assistant actually touched — for a tool that reads your mail and edits your calendar, 'show me exactly what it did' has no home today. Fixing both turns Himmy from 'works when I'm watching' into 'works for me, and I can see it did.'

**What it takes:** For scheduling: either a macOS launchd agent that ticks the same routines.py firing path headlessly, or keep Himmy resident in the menu bar so Quit/sleep doesn't kill the backend; and fix the catch-up so a missed daily fires at the next real slot, not instantly. The native notification primitive already exists in main.cjs (Notification is imported and wired) — just route routine results to it. For provenance: himmy already emits TOOL_CALLED/TOOL_COMPLETED events (cli.py:38 parses them but throws them away) — persist them per turn and add a plain-English 'Activity' panel. While here, add backend supervision (main.cjs startBackend returns before Python is listening and never respawns on crash) and stop the dozens of silent 'except: pass' from hiding failures the user must act on — chiefly the Google reconnect prompt.

## Build order (verified sequencing)
- BUILD FIRST: whole-workspace backup + safe restore (reliability #1 / distinctiveness #6). It is a prerequisite for every other move — each subsequent move (persisted RAG index, annotation ingest, corpus recs, learning loop) increases the volume of durable per-user state, and that state is currently unprotected with a destructive restore. Protect the data before you grow it.
- SECOND: persist the RAG index to disk via SqliteKnowledgeBackend (agent #2). This is the shared substrate for (a) feeding highlights/notes into RAG, (b) corpus-driven recommendations, and (c) the overnight analyst — all of which want a warm, durable, dedup'd index rather than re-embedding every launch. Doing it first means the annotation-ingest and recs moves write into a persistent store from day one.
- THIRD: feed highlights + notes into RAG (agent #1 / distinctiveness #2). Depends on the persisted index above (so re-ingesting annotations is incremental, not a full rebuild) and on the cache-key busting noted in the move. Once done, it directly unblocks the richer 'ask about this' context (product #5, which can then surface notes) and improves the analyst's library leg.
- FOURTH: corpus-driven recommendations (distinctiveness #1). Reuses the same embedder + persisted corpus vectors from the RAG-persistence step; building it after the index is durable avoids recomputing a corpus centroid on every poll.
- FIFTH (parallelizable, low-coupling): backend supervision/readiness (reliability #4), provenance trail (reliability #5), task due-dates (product #3), and global search (product #2). These are largely independent UI/Electron/endpoint work and can proceed once data safety (#1) is in place; global search should come after the notes surface exists if it is to include notes.
- DEFER until the substrate is proven: the iterative research council (agent #3) and the overnight analyst routine (distinctiveness #3). Both depend on a warm persisted index, corpus-aware recs, and a real web-search key (HIMMY_SEARCH_API_KEY) that is currently absent — wiring them on top of an ephemeral index or an empty web leg yields little. The cold-start/embed-off-loop fix should be attempted only inside himmy's KnowledgeBase (not re-attempting the reverted app-side warm).

## Verifier corrections (don't over-sell)
- ⚠️ RELIABILITY LENS overstates 'scheduler only runs while the app window is open / close the window stops automations'. PARTLY WRONG on macOS: main.cjs:151-152 window-all-closed only quits on NON-darwin; on macOS closing the window leaves the app (and backend, and the in-process AppScheduler) alive in the dock. before-quit (155-156) kills the backend only on full Quit. So the accurate statement is: automations stop on full Quit or Mac sleep, NOT on closing the window. The standalone-scheduler move is still valid for sleep/quit, but the framing is exaggerated.
- ⚠️ DISTINCTIVENESS + LEARNING LENS claims 'memory is store-only, there is no auto-injection adapter, Himmy forgets unless asked to recall' and 'register MemoryContextAdapter'. FALSE / ALREADY DONE: from_spec.py:321-344 appends MemoryContextAdapter(memory, top_k=spec.memory_top_k) as a context adapter whenever spec.memory is true; agent.yaml has memory: true + memory_top_k: 5. So memory IS auto-injected every turn. The AGENT LENS is correct here and explicitly flags this as 'better than the prior analysis implied' — the distinctiveness lens is repeating the very over-sell we were warned about.
- ⚠️ DISTINCTIVENESS LENS move #3 'add the macOS Notification (electron Notification confirmed unused)'. PARTLY WRONG: main.cjs:4 imports Notification and :23-25 has a working notify() using Notification.isSupported()/new Notification(...). The native-notification primitive already exists; the real gap is only WIRING routine results to it, not building it.
- ⚠️ RESEARCH.PY DOCSTRING is stale and misleads any reader: research.py:51-55 and :184 claim 'The server WARMS this index at startup (see server.py:_warm_rag_index)'. No such warm exists — server.py:222-226 explicitly documents the warm was TRIED (thread + process-pool) and REVERTED for stability. The reliability lens correctly says cold-start is unmitigated; but its proposed fix ('embed off the event loop + warm in background') is exactly what was already attempted and reverted because the GIL didn't release during the fastembed/ONNX embed — so the move as written is over-optimistic; the genuine fix must live in himmy's KnowledgeBase ingest (process pool / GIL-releasing), as the code comment itself notes.
- ⚠️ PRODUCT LENS move #7 / agent lens framing 'Mail is read-only, the agent cannot send'. The agent CAN send: agent.yaml:139 lists mail_send and agent_tools.py registers ActionsConnector with an approval-gated mail_send. The stale comments (agent.yaml:125-126 'gmail_send deliberately NOT listed'; agent_tools.py:26) contradict the actual allowlist. The product lens's point is narrowly about the UI MailRow being a read-only one-liner (true), but any claim that Himmy is mail-read-only end-to-end is wrong.
